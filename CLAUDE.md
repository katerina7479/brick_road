# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

`brick_road` is a Bevy-based planning-simulation desktop app. The core idea (see `brick_road_prd_v0.3.md`) is that **the schedule is an emergent output, not a maintained artifact**: the user defines work, dependencies, resources, and a calendar, and the app computes/visualizes the timeline. It's a 2D timeline editor built on Bevy 0.18 + bevy_egui 0.39, with all state persisted to a local SQLite file.

## Commands

```bash
cargo dev          # run with bevy/dynamic_linking for faster incremental rebuilds (alias for `cargo run --features dev`)
cargo run          # run a normal (statically linked) build
cargo test         # run all tests
cargo test calendar::   # run tests in one module (e.g. calendar)
cargo build        # compile without running
```

There is no CI, rustfmt.toml, or clippy config — use `cargo fmt` and `cargo clippy` defaults.

The app opens/creates `brick_road.db` (gitignored SQLite file) in the working directory on launch. Delete it to reset to a freshly seeded demo plan.

## Architecture

This is a **single-binary Bevy app with no custom plugins** — everything is wired directly in `main()` (`src/main.rs`) as a long list of Bevy systems and inserted resources. There is no plugin module; to understand the app, read the `App::new()...run()` chain in `main.rs`, which is the authoritative system schedule and ordering.

### The central data flow

```
brick_road.db ──load_model──▶ Model (Resource, the single source of truth)
                                 │
                    ┌────────────┼─────────────────┐
                    ▼            ▼                  ▼
              ScheduleAnalysis  Schedule        egui UI panels
              (violations,      (computed       (side_panel_ui edits Model)
               conflicts,        placement)            │
               critical path)        │                 ▼
                    │                 │       handle_block_drag/resize mutate Model
                    └────────┬────────┘                │
                             ▼                          ▼
                      sprites + gizmos          db::save_model (atomic, after every edit)
```

- **`Model`** (`model.rs`) is the one source of truth, a Bevy `Resource` holding `HashMap`s of all entities. UI and drag systems mutate it in place.
- **`Schedule`** (`schedule.rs`) is *derived* output from `Model` + the dependency graph (via `forward_pass`). It is recomputed on plan switch / auto-schedule, not every frame.
- **`ScheduleAnalysis`** (`analysis.rs`) is derived: dependency violations, resource conflicts, critical path, float. Recomputed only when `model.is_changed()`.
- **Persistence is auto-save**: there is no Save button. Every mutating UI action calls `db::save_model(&conn, &model)` immediately. When you add a new mutation path, you must add the `save_model` call too.

### Domain model (`model.rs`)

Read `model.rs` first — it defines the whole vocabulary. Key concepts:

- **World** — shared reality: the pool of resources a plan is evaluated against.
- **Plan** — a proposed future: `root_blocks`, `selected_variants` (which variant is active per block), resource `allocations`, and an optional `branch_start_day` (forked plans clamp block starts to ≥ this day).
- **WorkBlock** — a unit of effort with a three-point `Estimate` (optimistic/most_likely/pessimistic + confidence). A block with `variants` is a *choice point*, not a leaf; its own estimate is ignored and rolled up from the chosen variant's children.
- **Variant** — one alternative decomposition of a parent block into ordered `children`. Stores `block_positions` so child placements survive deactivation/reactivation.
- **Dependency** — `DependencyType` (FinishToStart/StartToStart/FinishToFinish/StartToFinish) plus `lag` (positive = delay, negative = lead).
- **ResourceBlock** + `AvailabilityTimeline` (segments of capacity over time); **ResourceAllocation** ties a block to a resource with an `allocation_factor`.
- **CalendarConfig** — anchors day 0 to a real `start_date`, defines `working_days_per_week` and `non_working_dates`. All timeline positions are integer **working days** (`type Day = i32`); convert to pixels with `day as f32 * PIXELS_PER_DAY`. Use `calendar.rs` (`date_to_day` / `day_to_date` / `first_working_day_of_month`) for any date arithmetic — do not hand-roll working-day math.

All entity IDs are opaque newtypes generated by the `id_newtype!` macro (`WorkBlockId`, `PlanId`, etc.). Never use raw `u64`; `Model::alloc_id` is the only ID source.

### Module map (`src/`)

| Module | Responsibility |
|---|---|
| `main.rs` | App assembly + system schedule; egui panels (`top_bar_ui`, `side_panel_ui`, resource view); grid/band/marker gizmos; fork-on-click. |
| `model.rs` | Domain entities + `Model` store + creation methods. Start here. |
| `db.rs` | SQLite schema, inline migrations, and `load_model` / `save_model`. |
| `schedule.rs` | `forward_pass` scheduler, `cascade_dependencies` (on-drag), critical-path / float (`analyze_user_placement`); `Schedule`, `ViewScope`, `TimelineViewMode` resources. |
| `graph.rs` | Builds the dependency DAG and detects cycles. |
| `analysis.rs` | Dependency-violation + resource-conflict detection feeding `ScheduleAnalysis`. |
| `blocks.rs` | Block sprite reconciliation, drag/resize/select/name-edit, dependency-edge drawing, undo, overlays (conflict/uncertainty/compare/past). The interaction-heavy module. |
| `labels.rs` | egui/world labels (block names, day numbers, period headers), nesting-depth + violation indicators. |
| `camera.rs` | 2D pan/zoom with exponential smoothing (`smooth_camera`), fit-to-view, keyboard nav. |
| `constants.rs` | Layout constants: `PIXELS_PER_DAY`, `ROW_HEIGHT`, `SIDE_PANEL_WIDTH`. |

### Two timeline views

`TimelineViewMode` toggles between **Task view** (blocks on a day×row timeline, default) and **Resource view** (one row per resource showing its allocations). Many render/interaction systems are gated by `.run_if(task_view_active)`; the resource view has its own `draw_resource_timeline` / `handle_resource_drag`. When adding a timeline-rendering system, decide which view(s) it belongs to and gate accordingly.

## Conventions & gotchas

- **System ordering is explicit and load-bearing.** `main.rs` uses `.before()`/`.after()`/`.chain()` extensively (e.g. analysis must run before sprite reconciliation). When adding a system, place it in the existing ordering rather than appending blindly, or you'll get a frame of stale visuals.
- **Change-detection guards.** Most derived systems early-return with `if !model.is_changed() { return; }` (or similar). Mutating a `ResMut<Model>` you don't actually change still trips `is_changed()` — only take `ResMut` when you will mutate.
- **The SQLite `Connection` is a `NonSend` resource** (rusqlite isn't `Send`). Access it with `NonSend<rusqlite::Connection>`; it cannot be used from a parallel system that requires `Send`.
- **DB migrations are inline `ALTER TABLE` statements** in `db::create_tables` that swallow "duplicate column name" errors (SQLite has no `ADD COLUMN IF NOT EXISTS`). Add new columns the same way; do not edit `CREATE_TABLES_SQL` for columns that must work on existing databases.
- **`save_model` is a full atomic rewrite** (delete join tables → upsert entities → delete stale rows → reinsert joins) in one transaction. Adding a new entity/relation means updating both `save_model` and `load_model`, plus the schema.
- **Colors are HDR linear RGB.** Values > 1.0 are intentional — they drive the bloom post-process (the camera uses `Hdr` + `Bloom` + `TonyMcMapface` tonemapping). The "today" marker and selection highlights rely on this.
- **Tests are inline `#[cfg(test)]` modules** (currently in `model.rs` and `calendar.rs`); there is no `tests/` directory. Put new unit tests next to the code they cover.
